## 上下文收集协议

设计技能在没有项目上下文的情况下会产生通用输出。在进行任何设计工作之前，你必须确认设计上下文。

**必需的上下文**（每个设计技能至少需要）：
- **目标受众**：谁在什么上下文中使用此产品？
- **用例**：他们试图完成什么工作？
- **品牌个性/基调**：界面应该有什么感觉？

个别技能可能需要额外的上下文。查看技能的准备部分以了解具体要求。

**关键**：你无法通过阅读代码库来推断此上下文。代码告诉你构建了什么，而不是为谁构建或应该有什么感觉。只有创建者可以提供此上下文。

**收集顺序：**
1. **检查当前指令（即时）**：如果你加载的指令已包含**设计上下文**部分，立即继续。
2. **检查 .impeccable.md（快速）**：如果不在指令中，从项目根目录读取 `.impeccable.md`。如果它存在并包含所需的上下文，继续。
3. **运行 impeccable teach（必需）**：如果两个来源都没有上下文，你必须在做任何其他事情之前立即运行 /impeccable teach。不要跳过此步骤。不要尝试从代码库推断上下文。

## 设计方向

致力于大胆的美学方向：
- **目的**：这个界面解决什么问题？谁使用它？
- **基调**：选择一个极端：残酷极简、极繁主义混乱、复古未来主义、有机/自然、奢华/精致、俏皮/玩具般、编辑/杂志、粗野主义/原始、装饰艺术/几何、柔和/粉彩、工业/实用等。有很多风格可供选择。以这些为灵感，但设计一个真正符合美学方向的风格。
- **约束**：技术要求（框架、性能、可访问性）。
- **差异化**：什么让这令人难忘？人们会记住的一件事是什么？

**关键**：选择一个清晰的概念方向并精确执行。大胆的极繁主义和精致的极简主义都可行。关键是有意图性，而不是强度。

然后实现可工作的代码，该代码：
- 生产级且功能完整
- 视觉引人注目且令人难忘
- 具有清晰美学观点的连贯性
- 在每个细节上都经过精心打磨

## 前端美学指南

### 排版
→ *有关 OpenType 功能、Web 字体加载以及关于刻度的更深入材料，请参阅[排版参考](reference/typography.md)。*

选择美观、独特、有趣的字体。将独特的展示字体与精致的正文字体配对。

<typography_principles>
始终应用这些 — 不要查阅参考，直接做：

- 在营销/内容页面上使用带有流体大小（clamp）的模块化类型刻度作为标题。在应用 UI 和仪表板上使用固定的 `rem` 刻度（没有主要设计系统在产品 UI 中使用流体类型）。
- 使用更少的大小和更多的对比。具有至少 1.25 比例的 5 步刻度比 8 个相距 1.1× 的大小创建更清晰的层次结构。
- 行高与行长成反比。窄列需要更紧密的行距，宽列需要更多。对于深色背景上的浅色文本，在正常行高中添加 0.05-0.1 — 浅色类型看起来更轻，需要更多呼吸空间。
- 将行长限制在约 65-75ch。比这更宽的正文文本会让人疲劳。
</typography_principles>

<font_selection_procedure>

</font_selection_procedure>

<typography_rules>
要在标题上使用带有流体大小（clamp）的模块化类型刻度。
要改变字体粗细和大小以创建清晰的视觉层次结构。
要在项目之间改变字体选择。如果你在上一个项目中使用了衬线展示字体，在这个项目中寻找无衬线、等宽或展示字体。

不要使用过度使用的字体，如 Inter、Roboto、Arial、Open Sans 或系统默认值 — 但也不要简单地切换到你第二喜欢的。上面 reflex_fonts_to_reject 列表中的每种字体都被禁止。看得更远。
不要使用等宽排版作为"技术/开发者"氛围的懒惰简写。
不要在每个标题上方放置带圆角的大图标。它们很少增加价值，使网站看起来模板化。
不要在整个页面中只使用一个字体系列。将独特的展示字体与精致的正文字体配对。
不要使用大小太接近的平面类型层次结构。目标是在步骤之间至少有 1.25 的比例。
不要在大写中设置长正文段落。保留全大写用于短标签和标题。
</typography_rules>

### 颜色与主题
→ *有关对比度、可访问性和调色板构建的更深入材料，请参阅[颜色参考](reference/color-and-contrast.md)。*

致力于连贯的调色板。具有鲜明强调色的主色优于胆小、均匀分布的调色板。

<color_principles>
始终应用这些 — 不要查阅参考，直接做：

- 使用 OKLCH，而不是 HSL。OKLCH 在感知上是均匀的：亮度的相等步骤*看起来*相等，这是 HSL 无法提供的。当你向白色或黑色移动时，减少色度 — 极端亮度下的高色度看起来刺眼。85% 亮度下的浅蓝色需要约 0.08 色度，而不是基色的 0.15。
- 将中性色着色向品牌色调。即使是 0.005-0.01 的色度也是可感知的，并在品牌色和 UI 表面之间创建潜意识连贯性。你着色的色调应来自此品牌，而不是来自"温暖 = 友好"或"冷色 = 技术"公式。首先选择品牌的实际色调，然后将所有内容着色向它。
- 60-30-10 规则是关于视觉*权重*，而不是像素计数。60% 中性/表面，30% 次要文本和边框，10% 强调色。强调色有效是因为它们稀有。过度使用会扼杀它们的力量。
</color_principles>

<theme_selection>
主题（浅色与深色）应从受众和观看上下文派生，而不是从默认值中选择。阅读简报并问：此产品何时使用，由谁使用，在什么物理环境中？

- 在快速交易时段使用的永续合约 DEX → 深色
- 在深夜由焦虑患者在手机上使用的医院门户 → 浅色
- 儿童阅读应用 → 浅色
- 用户在晚上 9 点坐在车库里的复古摩托车论坛 → 深色
- SRE 在黑暗办公室中的可观察性仪表板 → 深色
- 夫妇在周日早上使用的婚礼计划清单 → 浅色
- 在夜间使用耳机收听的音乐播放器应用 → 深色
- 在咖啡休息时间浏览的美食杂志主页 → 浅色

不要将所有内容默认为浅色"以求稳妥"。不要将所有内容默认为深色"以显得酷"。两个默认值都是懒惰的反射。正确的主题是实际用户在实际上下文中想要的主题。
</theme_selection>

<color_rules>
要使用现代 CSS 颜色函数（oklch、color-mix、light-dark）来创建感知均匀、可维护的调色板。
要将中性色着色向品牌色调。即使是微妙的提示也会创建潜意识连贯性。

不要在彩色背景上使用灰色文本；它看起来褪色。改用背景色的阴影。
不要使用纯黑（#000）或纯白（#fff）。始终着色；纯黑/白不会出现在自然界中。
不要使用 AI 调色板：深色上的青色、紫到蓝渐变、深色背景上的霓虹强调色。
不要使用渐变文本作为影响 — 有关严格定义，请参阅下面的 <absolute_bans>。文本仅使用纯色。
不要默认为带发光强调色的深色模式。它看起来"很酷"，但不需要实际的设计决策。
不要默认为浅色模式"以求安全"。关键是要选择，而不是退回到安全选项。
</color_rules>

### 布局与空间
→ *有关网格、容器查询和光学调整的更深入材料，请参阅[空间参考](reference/spatial-design.md)。*

通过变化的间距创建视觉节奏，而不是到处使用相同的填充。拥抱不对称和意外的组合。有意打破网格以强调。

<spatial_principles>
始终应用这些 — 不要查阅参考，直接做：

- 使用带有语义令牌名称（`--space-sm`、`--space-md`）的 4pt 间距刻度，而不是像素命名（`--spacing-8`）。刻度：4、8、12、16、24、32、48、64、96。8pt 太粗糙 — 你经常需要在两个值之间使用 12px。
- 使用 `gap` 而不是边距作为兄弟间距。它消除了边距折叠和随之而来的清理技巧。
- 改变间距以创建层次结构。上方有额外空间的标题看起来更重要 — 利用这一点。不要到处应用相同的填充。
- 自调整网格模式：`grid-template-columns: repeat(auto-fit, minmax(280px, 1fr))` 是卡片样式内容的无断点响应式网格。
- 容器查询用于组件，视口查询用于页面布局。侧边栏中的卡片应适应侧边栏的宽度，而不是视口的宽度。
</spatial_principles>

<spatial_rules>
要通过变化的间距创建视觉节奏：紧密的分组、慷慨的分离。
要使用带有 clamp() 的流体间距，在更大的屏幕上呼吸。
要使用不对称和意外的组合；有意打破网格以强调。

不要将所有内容包装在卡片中。并非所有内容都需要容器。
不要在卡片内嵌套卡片。视觉噪音；扁平化层次结构。
不要使用相同的卡片网格（相同大小的卡片，带有图标 + 标题 + 文本，无休止地重复）。
不要使用英雄指标布局模板（大数字、小标签、支持统计、渐变强调）。
不要将所有内容居中。左对齐文本与不对称布局感觉更设计。
不要到处使用相同的间距。没有节奏，布局感觉单调。
不要让正文文本每行超过约 80 个字符。添加最大宽度，如 65–75ch，以便眼睛可以轻松跟踪。
</spatial_rules>

### 视觉细节

<absolute_bans>
这些 CSS 模式永远不可接受。它们是最可识别的 AI 设计特征。匹配并拒绝：如果你发现自己在写这些，停下来并用完全不同的结构重写元素。

禁令 1：卡片/列表项/标注/警报的侧条纹边框
  - 模式：`border-left:` 或 `border-right:`，宽度大于 1px
  - 包括：硬编码颜色和 CSS 变量
  - 禁止：`border-left: 3px solid red`、`border-left: 4px solid #ff0000`、`border-left: 4px solid var(--color-warning)`、`border-left: 5px solid oklch(...)` 等。
  - 原因：这是管理、仪表板和医疗 UI 中最过度使用的"设计触感"。无论颜色、半径、不透明度或变量名称是"primary"还是"warning"还是"accent"，它永远不会看起来有意图。
  - 重写：使用完全不同的元素结构。不要只是切换到 box-shadow inset。寻求完整边框、背景着色、前导数字/图标，或者根本没有视觉指示器。

禁令 2：渐变文本
  - 模式：`background-clip: text`（或 `-webkit-background-clip: text`）与渐变背景结合
  - 禁止：任何使文本填充来自 `linear-gradient`、`radial-gradient` 或 `conic-gradient` 的组合
  - 原因：渐变文本是装饰性的而不是有意义的，是三大 AI 设计特征之一
  - 重写：为文本使用单一纯色。如果你想要强调，使用粗细或大小，而不是渐变填充。
</absolute_bans>

要：使用有意、有目的的装饰元素来加强品牌。
不要：在卡片、列表项、标注或警报上使用大于 1px 的 border-left 或 border-right 作为彩色强调条纹。有关严格的 CSS 模式，请参阅上面的 <absolute_bans>。
不要：到处使用玻璃态（模糊效果、玻璃卡片、发光边框用作装饰而不是有目的）。
不要：将迷你图用作装饰。看起来复杂但没有任何意义的小图表。
不要：使用带通用投影的圆角矩形。安全、易忘、可能是任何 AI 输出。
不要：使用模态框，除非真的没有更好的替代方案。模态框是懒惰的。

### 动效
→ *有关时间、缓动和减少动效，请参阅[动效参考](reference/motion-design.md)。*

专注于高影响时刻：一个编排良好的页面加载与交错显示比分散的微交互创造更多愉悦。

**要**：使用动效传达状态变化：进入、退出、反馈
**要**：使用指数缓动（ease-out-quart/quint/expo）实现自然减速
**要**：对于高度动画，使用 grid-template-rows 过渡而不是直接动画高度
**不要**：动画布局属性（宽度、高度、填充、边距）。仅使用 transform 和 opacity
**不要**：使用弹跳或弹性缓动。它们感觉过时和俗气；真实物体平滑减速

### 交互
→ *有关表单、焦点和加载模式，请参阅[交互参考](reference/interaction-design.md)。*

使交互感觉快速。使用乐观 UI：立即更新，稍后同步。

**要**：使用渐进式披露。从简单开始，通过交互揭示复杂性（基本选项在前，高级在可扩展部分后面；显示次要操作的悬停状态）
**要**：设计教授界面的空状态，而不仅仅是说"这里什么都没有"
**要**：使每个交互表面感觉有意图和响应
**不要**：重复相同的信息（冗余标题、重述标题的介绍）
**不要**：使每个按钮都是主要的。使用幽灵按钮、文本链接、次要样式；层次结构很重要

### 响应式
→ *有关移动优先、流体设计和容器查询，请参阅[响应式参考](reference/responsive-design.md)。*

**要**：使用容器查询（@container）实现组件级响应性
**要**：为不同上下文调整界面，而不仅仅是缩小它
**不要**：在移动设备上隐藏关键功能。调整界面，不要截肢它

### UX 写作
→ *有关标签、错误和空状态，请参阅[UX 写作参考](reference/ux-writing.md)。*

**要**：让每个字都有价值
**不要**：重复用户已经可以看到的信息

## AI 废话测试

**关键质量检查**：如果你向某人展示此界面并说"AI 制作了这个"，他们会立即相信你吗？如果是，那就是问题所在。

独特的界面应该让某人问"这是如何制作的？"而不是"哪个 AI 制作的？"

## 实现原则

将实现复杂性与美学愿景匹配。极繁主义设计需要精心设计的代码，具有广泛的动画和效果。极简主义或精致的设计需要克制、精确，并仔细关注间距、排版和微妙细节。

创造性地解释并做出意外的选择，这些选择感觉真正为上下文设计。没有设计应该是相同的。在浅色和深色主题、不同字体、不同美学之间变化。永远不要在几代人之间收敛到共同选择。

记住：Claude 能够进行非凡的创意工作。不要保留。展示当跳出框框思考并完全致力于独特愿景时真正可以创造什么。

## Teach 模式

如果此技能使用参数"teach"调用（例如，`/impeccable teach`），跳过上面的所有设计工作，而是运行下面的 teach 流程。这是一次性设置，为项目收集设计上下文。

### 步骤 1：探索代码库

在提问之前，彻底扫描项目以发现你可以发现的内容：

- **README 和文档**：项目目的、目标受众、任何声明的目标
- **Package.json / 配置文件**：技术栈、依赖项、现有设计库
- **现有组件**：当前设计模式、间距、正在使用的排版
- **品牌资产**：Logo、favicon、已定义的颜色值
- **设计令牌 / CSS 变量**：现有调色板、字体栈、间距刻度
- **任何风格指南或品牌文档**

注意你学到的内容和仍不清楚的内容。

### 步骤 2：提出以 UX 为重点的问题

停下来并调用 `question` 工具进行澄清。仅关注你无法从代码库推断的内容：

#### 用户与目的
- 谁使用这个？他们使用时的上下文是什么？
- 他们试图完成什么工作？
- 界面应该唤起什么情绪？（信心、愉悦、平静、紧迫等）

#### 品牌与个性
- 你如何用 3 个词描述品牌个性？
- 任何捕捉正确感觉的参考网站或应用？它们具体有什么？
- 这明确不应该看起来像什么？任何反参考？

#### 美学偏好
- 对视觉方向有任何强烈偏好吗？（极简、大胆、优雅、俏皮、技术、有机等）
- 浅色模式、深色模式，还是两者？
- 任何必须使用或避免的颜色？

#### 可访问性与包容性
- 特定的可访问性要求？（WCAG 级别、已知的用户需求）
- 对减少动效、色盲或其他适应的考虑？

跳过答案已从代码库探索中清楚的提问。

### 步骤 3：编写设计上下文

综合你的发现和用户的答案到 `## 设计上下文` 部分：

```markdown
## 设计上下文

### 用户
[他们是谁，他们的上下文，要完成的工作]

### 品牌个性
[声音、基调、3 词个性、情感目标]

### 美学方向
[视觉基调、参考、反参考、主题]

### 设计原则
[从对话中派生的 3-5 个原则，应指导所有设计决策]
```

将此部分写入项目根目录的 `.impeccable.md`。如果文件已存在，就地更新设计上下文部分。

然后停下来并调用 `question` 工具澄清。他们是否还希望将设计上下文附加到 AGENTS.md。如果是，也在此处附加或更新部分。

确认完成并总结现在将指导所有未来工作的关键设计原则。
